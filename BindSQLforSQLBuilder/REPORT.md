- 状態: 新規
- 分類: バグ
- タイトル: 位置バインドの欠番で番号ではなく出現順に `?` が置換される
- 種別: 事実
- 要約: `[2] = 200` のような欠番入力で、2番目の `?` ではなく1番目の `?` に `200` が入る。変換結果が静かに誤るため、実行SQLの信頼性を壊す。
- 証拠: `BindSQLforSQLBuilder.html:677-707` の `_parseBindVariables` が添字付き配列ではなく値配列へ落としている。`BindSQLforSQLBuilder.html:729-737` の `_mergeSqlAndBinds` は配列先頭から順番に `?` を消費する。操作手順: `SELECT ? AS first, ? AS second` と `[2] = 200` を貼ると、2026-04-23 の Playwright 再現で `200 AS first` と出力された。
- 内容: 事実: 位置番号を保持せず `_positional.map((_entry) => _entry._value)` に変換しているため、欠番や疎なログを正しく扱えない。推論: 運用で一部のバインドだけ抜けたログや、複数行からのコピー漏れが起きると、誤ったSQLをそのままDBへ流す危険がある。
- ユーザー影響: 結果SQLの意味が変わる。調査時間の増加ではなく、誤調査や誤更新につながる。

- 期待効果: 位置バインドの再現性が上がり、出力SQLを目視確認なしでも信頼しやすくなる。
- 緊急度: 高
- 実装コスト: 小
- 確信度: 高
- 次の行動: 位置バインドを `index -> value` の連想構造で保持し、`?` 置換時に1始まりの添字で参照する。`[2]=200`、`1=100,3=300`、欠番ありのケースを回帰確認項目に追加する。
