#### Issue: 20260423-01
- 状態: 新規
- 分類: バグ
- タイトル: 位置バインドの欠番で番号ではなく出現順に `?` が置換される
- 種別: 事実
- 要約: `[2] = 200` のような欠番入力で、2番目の `?` ではなく1番目の `?` に `200` が入る。変換結果が静かに誤るため、実行SQLの信頼性を壊す。
- 証拠: `BindSQLforSQLBuilder.html:677-707` の `_parseBindVariables` が添字付き配列ではなく値配列へ落としている。`BindSQLforSQLBuilder.html:729-737` の `_mergeSqlAndBinds` は配列先頭から順番に `?` を消費する。操作手順: `SELECT ? AS first, ? AS second` と `[2] = 200` を貼ると、2026-04-23 の Playwright 再現で `200 AS first` と出力された。
- 内容: 事実: 位置番号を保持せず `_positional.map((_entry) => _entry._value)` に変換しているため、欠番や疎なログを正しく扱えない。推論: 運用で一部のバインドだけ抜けたログや、複数行からのコピー漏れが起きると、誤ったSQLをそのままDBへ流す危険がある。
- ユーザー影響: 結果SQLの意味が変わる。調査時間の増加ではなく、誤調査や誤更新につながる。
- 制約環境での影響: 監視・本番調査のように再実行回数を減らしたい現場ほど、1回の誤変換のコストが高い。
- 期待効果: 位置バインドの再現性が上がり、出力SQLを目視確認なしでも信頼しやすくなる。
- 緊急度: 高
- 実装コスト: 小
- 確信度: 高
- 次の行動: 位置バインドを `index -> value` の連想構造で保持し、`?` 置換時に1始まりの添字で参照する。`[2]=200`、`1=100,3=300`、欠番ありのケースを回帰確認項目に追加する。

#### Issue: 20260423-03
- 状態: 新規
- 分類: バグ
- タイトル: 1行ログのスマートクォート値にカンマが入るとパラメータ分割が壊れる
- 種別: 事実
- 要約: `“A,B”` のような値を1行ログで与えると、カンマで途中分割されて値が `‘“A’` までに欠ける。コード上ではスマートクォートを一部サポートしているが、分割器が追随していない。
- 証拠: `BindSQLforSQLBuilder.html:437-480` の `_splitParameterEntries` は引用符判定を `'` と `"` に限定している。一方で `BindSQLforSQLBuilder.html:590-592` の `_normalizeBindValue` は `“”` を文字列値として扱う。操作手順: `sql=[select * from t where name=?] parameters=[1=“A,B”]` を貼ると、2026-04-23 の Playwright 再現で `name = '“A'` と出力された。
- 内容: 事実: 解析器内で引用符の定義が不統一で、同じ入力形式の前半と後半で扱いが違う。推論: Windowsのドキュメントやチャットからログを経由貼り付けしたときに、文字種変換で混入したスマートクォートが原因で誤変換しやすい。
- ユーザー影響: 文字列条件の一部が失われ、検索条件が変わる。見た目では気付きにくい。
- 制約環境での影響: IME変換や社内文書経由のコピーが多い環境ほど混入しやすい。
- 期待効果: 貼り付け元が混在しても変換の安定性が上がる。
- 緊急度: 高
- 実装コスト: 小
- 確信度: 高
- 次の行動: `_splitParameterEntries` の引用符判定を `“”` まで広げ、スマートクォート付きの1行ログを変換回帰ケースに追加する。

#### Issue: 20260423-04
- 状態: 新規
- 分類: バグ
- タイトル: バッククォート識別子内の予約語が強制大文字化される
- 種別: 事実
- 要約: MySQL系で使うバッククォート識別子が保護されておらず、`` `select` `` が `` `SELECT` `` に変わる。方言依存だが、識別子の大小が意味を持つ運用では危険。

- 緊急度: 中
- 実装コスト: 小
- 確信度: 高
- 次の行動: SQL保護セグメントにバッククォート識別子を追加し、キーワード大文字化と改行整形が識別子内部へ入らないことを確認する。
