#### ISS-002
- 分類: バグ / UX
- タイトル: `localStorage` 保存失敗時に起動・操作が例外で止まる
- 種別: 事実
- 要約: `_saveState()` は `localStorage.setItem()` を直接呼び、例外を捕捉していない。保存不可環境では初期表示末尾の `_saveState()` だけで例外になり得る。
- 証拠: `MustKnowCodingEnglish.html` 9439-9442行目の `_saveState()` は `try/catch` なし。10300行目で起動直後に `_saveState()` を実行。検証では `localStorage.setItem()` が `quota exceeded` を投げるスタブ環境で、インラインスクリプト実行が `quota exceeded` で停止した。
- 内容: プライベートブラウズ、容量上限、ブラウザ設定、管理端末の制限で `localStorage` 書き込みが失敗すると、保存状態表示や学習継続以前にアプリ操作が崩れる可能性がある。ローカル完結ツールほど保存不可時の退避表示が必要。
- ユーザー影響: 学習開始前に画面が壊れる、または回答・リセット・テーマ変更後に操作結果が反映されない。
- 期待効果: 保存不可時でも「一時利用は可能」「保存できない理由が分かる」状態にでき、制約端末での導入リスクを下げられる。
- 緊急度: 高
- 実装コスト: 小
- 確信度: 高
- 次の行動: `_saveState()` を `try/catch` 化し、失敗時は `_runtime._saveError` とフッター文言に「保存できません。ブラウザ設定または容量を確認してください」を出す。起動時に保存を強制する必要があるかも見直す。
- スコア: ユーザー被害5 / 購入理由への寄与4 / 実装しやすさ4 / 壊しにくさ4 / 保守性寄与4 = 21

- 種別: 事実
- 要約: 設定モーダルは `role="dialog"` と `aria-modal="true"` を持つが、開いた時の初期フォーカス、閉じた後のフォーカス復帰、開いている間のフォーカストラップが実装されていない。

- ユーザー影響: キーボード操作や支援技術利用時に、設定を閉じた後の現在位置が分かりにくい。

- 緊急度: 中
- 実装コスト: 小
- 確信度: 高
- 次の行動: 開いたトリガー要素を `_runtime._settingsTrigger` に保持し、初期フォーカスを閉じるボタンへ移動。Tab循環とclose時のフォーカス復帰を追加する。

- 確信度: 中

### 4. 上位3件
- 1位: ISS-002 `localStorage` 保存失敗時に起動・操作が例外で止まる
  - 選定理由: ユーザー被害が5で、主要フローの起動・回答・テーマ変更・CSV出力の前提になる保存処理に直結する。
  - 今やるべき理由: ローカル完結を売りにするツールでは、保存できない環境での説明がないと信頼を失う。修正範囲は `_saveState()` とフッター表示中心で小さい。
  - 実装イメージ: `_saveState()` に `try/catch` を追加し、失敗時はメモリ上の状態だけで継続。`_renderFooter()` で保存失敗メッセージを出す。
  - 想定UI: フッターに「保存状態: このブラウザでは保存できません。一時利用中です。」を表示。
  - 想定ユースケース: 社内端末のブラウザ設定でlocalStorageがブロックされている状態で、学習カードだけ確認する。
  - リスク: 保存失敗時もユーザーが学習できるため、後で履歴が残っていないことに気づく可能性がある。文言で「一時利用中」を明示する。
  - 着手順: `_runtime._saveError` を追加 -> `_saveState()` をtry/catch化 -> `_renderFooter()` の表示分岐 -> 保存失敗スタブで確認。
  - 受け入れ条件: `localStorage.setItem` が例外を投げても初期表示処理が例外終了しない。保存不可メッセージがフッターに出る。
- 2位: ISS-001 「外部通信なし」とGoogle Fonts/CDN参照が矛盾している
  - 選定理由: 購入理由への寄与が5で、制約環境における販売価値とREADMEの信頼性に直結する。
  - 今やるべき理由: 変更はフォント/アイコン依存と説明文の整合が中心で、先に方針を決めると以後の訴求がぶれない。
  - 実装イメージ: 本体のMaterial Symbols参照をなくし、`code` アイコンを既存faviconと同系統のインラインSVGにする。READMEは「外部通信なし」を完全オフラインの意味に合わせる。

  - 想定ユースケース: ネットワーク遮断された研修PCでHTMLファイルを直接開く。
  - リスク: アイコンフォントの置換で視覚印象が少し変わる。SVGサイズと色を既存CSS変数に合わせる。
  - 着手順: 外部フォント依存箇所を確認 -> ヘッダーアイコンをSVG化 -> READMEの「外部通信なし」を実挙動に合わせる -> オフライン相当で表示確認。
  - 受け入れ条件: `MustKnowCodingEnglish.html` 本体に `fonts.googleapis.com` / `fonts.gstatic.com` 参照が残らない。READMEのローカル完結説明と実HTMLが矛盾しない。
- 3位: ISS-003 全800語の個別例文が未投入で学習価値が伝わりにくい

  - 優先度: 中

### 6. 訴求・販売改善案
- 想定ユーザー: 日本語話者の新人/若手エンジニア、英語のPRレビュー・エラー文・仕様書に苦手意識がある開発者、社内研修で短時間の英語復習を配りたい担当者。
- 解決している具体的な痛み: コーディングで頻出する英単語を、一般英語教材ではなく開発文脈で復習したい。社内端末で外部サービスに学習データを出したくない。短時間で苦手語だけを復習したい。
- 今でも刺さる強み: 800語をレベル別に収録。検索、ステータス、4択テスト、結果、優先復習、CSV出力が1HTML内にある。進捗がlocalStorageに残る。
- 伝わっていない価値: 「なぜ800語なのか」「どの現場文脈で使う語なのか」「オフライン/社内端末で使える範囲はどこまでか」「テスト後に何分短縮できるか」がREADMEにない。
- 買う理由として弱い点: 実務例文が未投入のため、無料の単語表との差が見えにくい。バックアップ/復元やセッション履歴出力がなく、本番利用の安心感が弱い。
- READMEに追加すべき説明: 対象ユーザー、3分の利用手順、保存されるデータと外部通信の正確な説明、制約環境での使い方、CSVで出せる内容、現在の制限。
- LPに追加すべき訴求: 「PRレビュー・エラー対応・ドキュメントで見る英語を1HTMLで復習」「学習データを外部サービスへ送らない」「テスト後に苦手語へ戻れる」「研修PCに配りやすい」。
- スクリーンショットで見せるべき場面: 検索で単語を絞る画面、実務例文つきカード、4択回答後の正誤フィードバック、結果画面の優先復習ワード、CSV出力後のファイル例。
- 無料版で見せるべき体験: 100語程度、検索、学習カード、5問テスト、直近結果、数語の実務例文。良さは伝わるが研修や本番復習には語数・履歴出力が不足する状態にする。
- 有料版でしか得られない体験: 800語、実務例文パック、10問テスト、セッションCSV、バックアップ/復元、苦手語の復習導線、社内配布向けの完全オフライン版。
- 制約のある現場で使える理由: 1HTMLで配布できる。学習データはブラウザ内保存にできる。外部通信を完全排除するならフォント/アイコンCDNを外す必要がある。
- 他手段より楽な理由: 辞書、Anki、スプレッドシートを往復せず、検索・意味確認・4択・復習対象確認を同じ画面内で完結できる。
- 導入直後に感動できる体験があるか: 現状は「800語が入っている」ことは分かるが、例文が汎用テンプレートのため感動は弱い。最初の30語に実務例文を入れると、導入直後の価値判断が改善する。
- 強み: 1ファイルSPA、800語、レベル/ステータス/アルファベット検索、テストと結果の往復、localStorage保存。
- 弱み: 外部通信説明の矛盾、例文未投入、復習主CTAの弱さ、履歴CSV/バックアップ不足。
- 使う理由: 外部サービスに学習データを出さず、開発文脈の頻出英語だけを短時間で復習するため。
- デモ版/有料版の切り分け案: デモ版は100語と5問テストで体験を見せる。有料版は800語、実務例文、履歴持ち出し、バックアップ、完全オフラインで「本番利用・再利用・安心」を提供する。
