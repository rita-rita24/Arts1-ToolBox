# REPORT

## 2026-04-26
### 0. エグゼクティブサマリー
- 今回の結論: LocalNotion は単一 HTML で多機能だが、保存の信頼性、共有後の再開、完全オフライン運用の3点に実務上の不安が残る。
- 最優先課題: 書き出した HTML を再度開いたときに前回編集内容を確実に再開できるようにすること。
- 次回の狙い: 保存経路を「通常編集」「大容量画像」「HTML 書き出し後の再開」の3ケースで壊れにくくすること。
- 今回の監査では実装は行っていない: はい。

### 1. 今回の監査対象
- 対象ファイル: `README.md`, `LocalNotion.html`
- 対象画面: サイドバー、ページ編集画面、検索モーダル、ダイアログ、画像ブロック、HTML 書き出し導線
- 主な利用フロー: 初回表示 → ページ作成 → ブロック編集 → 画像追加 → 検索 → HTML 書き出し → 再オープン
- 想定ユーザー: ネット接続に依存せずローカルでメモ・下書き・簡易ドキュメントを管理したい個人ユーザー
- ツールの主目的: 単一 HTML で動作する Notion 風のローカルエディタを提供すること
- 確認した観点: 保存/読込、書き出し、外部依存、状態遷移、誤操作耐性、アクセシビリティ、価値訴求、制約環境での運用適性

#### Issue: 20260426-01
- 状態: 新規
- 分類: バグ
- タイトル: 書き出した HTML を再オープンすると前回編集内容を復元できない可能性がある
- 種別: 推論
- 要約: 書き出し時に埋め込んだ `embedded-state` が起動時に常に優先される。書き出した HTML を開いて編集しても、次回起動時に書き出し時点の状態へ戻る恐れがある。
- 証拠: 対象ファイル `LocalNotion.html`; 関数 `#export-btn` の書き出し処理と `init`; 画面 `アプリをHTMLとして保存`; 操作手順 `HTMLを書き出す → 書き出したHTMLを開く → 編集する → 閉じる → 同じHTMLを再度開く`; 該当箇所 `2668-2701`, `3046-3058`
- 内容: 書き出し処理はその時点の `state` を `embedded-state` に埋め込むだけで、以後の編集でファイル自体は更新されない。一方で起動時は `IndexedDB` より `embedded-state` を優先しているため、ファイルを再オープンした際に直近編集より古い埋め込み状態が採用される設計になっている。
- ユーザー影響: HTML をバックアップ兼持ち運び手段として信頼できず、再開時に編集が消えたと認識する可能性が高い。
- 制約環境での影響: オフライン配布や USB 持ち運び前提の運用では、書き出しファイルが主たる保存媒体になるため影響が大きい。
- 期待効果: 書き出し後の再開信頼性が上がり、単一 HTML 配布の価値が実用レベルになる。
- 緊急度: 高
- 実装コスト: 中
- 確信度: 中
- 次の行動: 起動時の読込優先順位を見直し、同一ファイルでの再開時は永続化済み状態を優先する条件を追加する。

#### Issue: 20260426-04
- 状態: 新規
- 分類: バグ
- タイトル: `localStorage` への退避保存が復元経路に戻ってこない
- 種別: 推論
- 要約: 保存失敗時は `localStorage` に退避するが、読込側は `IndexedDB` のオープン自体が失敗した場合しか `localStorage` を見ない。退避データが実質的に孤立する経路がある。
- 証拠: 対象ファイル `LocalNotion.html`; 関数 `_doSave`, `loadStateAsync`; 画面 `保存/再起動`; 操作手順 `IndexedDB保存失敗 → localStorage退避成功 → 次回起動でIndexedDBは開けるが空`; 該当箇所 `1208-1226`
- 内容: `_doSave` は `IndexedDB` 保存失敗時にのみ `localStorage` へ落とす。一方 `loadStateAsync` は `openDB()` が成功した時点で `localStorage` を参照しないため、退避成功分を再読込できない条件が残る。
- ユーザー影響: 一度は退避できた編集内容が、次回起動時に見つからず消えたと受け取られる。
- 制約環境での影響: ブラウザ設定や拡張機能で `IndexedDB` が不安定な端末ほど、復旧できない確率が上がる。
- 期待効果: 退避保存が本当に復旧策として機能する。
- 緊急度: 高
- 実装コスト: 小
- 確信度: 中
- 次の行動: `IndexedDB` から値が取れなかった場合にも `localStorage` を確認し、存在時は `IndexedDB` へ再移行する。

#### Issue: 20260430-03
- 状態: 新規
- 分類: バグ
- タイトル: `outerHTML` フォールバック書き出しが一時 UI 状態まで保存する可能性がある
- 種別: 推論
- 要約: `fetch(selfUrl)` が失敗したときは live DOM の `document.documentElement.outerHTML` をそのまま保存している。サイドバー折りたたみ、モーダル表示、インラインツールバー、コンテキストメニューなどの一時状態が書き出し HTML に混ざる余地がある。

- 内容: フォールバックは「現在見えている DOM」を複製する方式で、初期テンプレートへ state だけを差し込む方式ではない。`hidden` 属性の外れたオーバーレイ、開いたままのメニュー、折りたたまれたサイドバーなどがそのまま保存されると、再オープン時の第一印象や説明責任が崩れる。今回の監査では `fetch(selfUrl)` 失敗環境での実ブラウザ再現までは行っていないため、影響範囲はコード証拠ベースの推論である。
- ユーザー影響: 配布先で開いた直後の画面が編集途中の姿に見えると、「保存物」としての信頼が下がる。
- 制約環境での影響: 単一 HTML を `file://` で配る、あるいは閉域端末で開く運用ほどフォールバック経路に入りやすい。
- 期待効果: 書き出しファイルの見た目と初期状態が安定し、持ち運びや共有の説明がしやすくなる。
- 緊急度: 中
- 実装コスト: 中
- 確信度: 中
- 次の行動: 書き出し元を常に静的テンプレートへ寄せるか、`outerHTML` フォールバック前にオーバーレイ・メニュー・選択状態を既定値へ戻す正規化処理を追加する。
